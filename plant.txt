## 🎯 技术栈确定

| 层        | 选型                                  | 理由                                                         |
|-----------|---------------------------------------|--------------------------------------------------------------|
| 后端框架  | **FastAPI**                           | 异步支持好，自动生成 Swagger 文档，适合 API 服务              |
| AI 框架   | **LangChain** / **LangChain4j**（Python版） | 你已学过，封装了 RAG 全流程                                |
| LLM       | OpenAI / 通义千问 / DeepSeek          | 前期可用便宜的 API 调试，后期可切换本地模型                    |
| 向量存储  | **Chroma** 或 **FAISS**               | 轻量级，无需额外部署服务，本地开发友好；正式环境可换 Milvus 等 |
| 文档处理  | LangChain 自带的 Loader + Text Splitter | 内置 PDF、Word、TXT 等加载器，省时                          |
| 部署      | Docker + Uvicorn                      | 一键打包，面试时方便演示                                     |

> 这里首推 **Chroma**，因为它是一个嵌入式向量数据库，安装只要 `pip install chromadb`，非常轻便，和 LangChain 集成完美。FAISS 也是类似选择。

## 🗓️ 第1周：跑通最简 RAG 链路（文档 → 检索 → 生成）

**目标**：上传一个 TXT/PDF，提问，返回基于文档的答案。

### 环境准备
- Python 3.10+
- 虚拟环境：`python -m venv venv`
- 核心依赖（`requirements.txt`）：
  ```
  fastapi
  uvicorn
  langchain
  langchain-openai  # 如果用 OpenAI
  chromadb
  pypdf           # 处理 PDF
  python-dotenv
  ```

### 任务清单
| 天数 | 任务 | 关键代码/产出 |
|------|------|---------------|
| Day 1 | 创建 FastAPI 项目骨架，配置环境变量（OpenAI Key），安装依赖 | `main.py` 能启动，`/health` 接口返回 200 |
| Day 2 | 实现文件上传接口，接收 PDF/TXT，使用 LangChain Loader 读取文本 | `/upload` API，返回提取的文本内容（先不切分） |
| Day 3 | 实现文本切分（RecursiveCharacterTextSplitter）+ 向量化（OpenAIEmbeddings）+ 存入 Chroma | 打印向量数量，确认存储成功 |
| Day 4 | 实现问答接口：用户问题 → 向量检索 → 拼接上下文 → 调用 LLM 回答 | `/chat` API，返回 `{"question":"...", "answer":"..."}` |
| Day 5 | 联调与简单的错误处理（文件不存在、解析失败等） | 用 Postman 测试完整链路 |
| Day 6 | 编写一个最小前端（或直接用 Swagger UI 演示）并录制演示视频 | 项目 README 中嵌入截图或 GIF |
| Day 7 | 代码整理，加入详细注释，为下周优化做准备 | 项目提交 Git |

**Day 4 核心代码片段**（让你心里有底）：
```python
from langchain.chains import RetrievalQA
from langchain.vectorstores import Chroma
from langchain_openai import ChatOpenAI, OpenAIEmbeddings

# 初始化向量数据库（假设之前已存入文档）
embeddings = OpenAIEmbeddings()
vectordb = Chroma(persist_directory="./chroma_db", embedding_function=embeddings)

# 构建问答链
qa_chain = RetrievalQA.from_chain_type(
    llm=ChatOpenAI(model="gpt-3.5-turbo"),
    retriever=vectordb.as_retriever(search_kwargs={"k": 3}),
    return_source_documents=True
)

# 提问
result = qa_chain("公司的年假政策是什么？")
print(result["result"])  # 答案
print(result["source_documents"])  # 引用来源
```

## 🗓️ 第2周：提升质量——HyDE、重排、多轮对话

**目标**：让答案更准，能追问，更像一个“企业级助手”。

### 需要实现的优化点

| 优化项      | LangChain 实现方式                                                                 | 效果                       |
|-------------|-----------------------------------------------------------------------------------|----------------------------|
| **HyDE**    | 使用 `HypotheticalDocumentEmbedder`（或手写：先生成假答案再检索）                  | 提高召回率，解决问题与文档表述不一致的问题 |
| **Rerank**  | 集成 `CohereRerank` 或 `bge-reranker`，在检索后加一步重排序                       | 把真正相关的片段放到最前面 |
| **多轮记忆**| 使用 `ConversationalRetrievalChain`，它会自动带上对话历史                         | 支持“那第二条呢？”这样的追问 |
| **来源引用**| 在返回答案时带上文档名称和段落位置，增强可信度                                     | 面试时展示“可解释AI”       |

**本周结束标志**：你可以在一个对话中连续提问，系统能记住上下文；答案质量明显比第一周高，引用的片段精准。

## 🗓️ 第3周：工程化封装 & 准备面试“故事”

**目标**：让项目从“脚本”变成“工程”，并打磨好 STAR 话术。

### 工程化动作
1. **配置管理**：使用 Pydantic Settings 管理 API Key、模型名等配置。
2. **结构化日志**：用 Loguru 或标准 logging 记录每次问答的耗时、检索文档数。
3. **Docker 化**：编写 `Dockerfile` 和 `docker-compose.yml`，把 FastAPI 应用和 Chroma 数据打包。
4. **API 文档**：FastAPI 自带 Swagger，做好接口描述、示例数据即可。
5. **简单压测**：用 `locust` 或 `wrk` 模拟 10 个并发用户，观察延迟，作为简历量化数据。

### 简历 STAR 再适配（Python 版本）
- **S**：企业内部文档分散，员工查找信息效率低。
- **T**：构建智能问答系统，支持自然语言秒级检索答案。
- **A**：
  - 使用 **LangChain + Chroma** 搭建 RAG 管道，实现文档自动分段、向量化与相似性检索。
  - 集成 **HyDE** 与 **重排序** 机制，将 Top3 答案准确率提升至 **85%** 以上。
  - 基于 **FastAPI** 提供 RESTful 服务，容器化部署，并编写全套接口文档。
- **R**：查询耗时降低 90%，支持 5 个部门知识库空间，系统在 100 并发下响应时间 < 2s（根据实测填真实数字）。

## 🔁 与 Java 方案的对比（面试时可能被问）

| 对比维度   | Java 方案 (Spring AI)            | Python 方案 (LangChain + FastAPI) |
|------------|----------------------------------|----------------------------------|
| 生态成熟度 | 较新，但发展快                   | **非常成熟**，教程/方案多       |
| 开发效率   | 中，依赖配置相对繁琐             | **快**，尤其数据处理和 AI 部分  |
| 性能       | 天生高并发，适合大规模服务        | FastAPI 异步性能也很好          |
| 企业化集成 | Spring 生态，易与微服务整合       | Python 在数据/AI 领域更占优    |
| 学习曲线   | 需熟悉 Spring Boot 全家桶        | 更平缓，适合快速原型           |

**你的回答模板**：“我选择 Python 是因为这个项目的核心是 AI 能力落地，LangChain 生态能让我更快迭代 RAG 策略。当需要与现有 Java 微服务打通时，我会把 Python 服务封装成 API 网关调用的单独模块。”
