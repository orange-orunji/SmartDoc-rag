
# 智能办公助手 — 简历面试问题准备

> 基于 `RAG_Personal` 项目（FastAPI + LangChain Agent + 混合检索 + RabbitMQ + Redis）梳理的面试高频问题，覆盖项目介绍、RAG 原理、工程架构、Agent 设计、缓存策略、消息队列等维度。

---

## 一、项目整体介绍

### Q1：请简单介绍一下这个项目
**考察点**：项目概述能力、业务理解

**回答要点**：
- 这是一个**企业级 AI 办公助手**，面向知识管理与办公自动化场景
- 核心能力：知识库文档检索问答、AI 报告生成、文件格式转换、邮件发送
- 技术栈：FastAPI + LangChain Agent + Chroma 向量库 + RabbitMQ + Redis
- 特色：**三阶混合检索（HyDE + 向量 + BM25 + Rerank）**、Agent 自主决策（6 个 Function Calling 工具）、双层缓存、异步文档处理、SSE 流式对话

### Q2：这个项目的最大技术亮点是什么？
**考察点**：对项目的深度理解

**回答要点**：
- **HyDE（假设文档嵌入）检索增强**：用 LLM 先生成"假想答案"再检索，桥接用户口语化提问与文档正式用语之间的语义鸿沟，召回率从 53% 提升到 67%
- **Function Calling Agent**：6 个工具自主决策调用，从检索到报告生成到邮件发送形成完整闭环
- **双层缓存**：MD5 精确匹配（Redis）+ 语义相似度匹配（内存向量），缓存命中延迟从 ~20s 降至 ~0.01s
- **量化评估体系**：内置 Recall@K、MRR 自动化评测脚本，多种检索策略 A/B 对比

### Q3：项目中有哪些你觉得可以改进的地方？
**考察点**：批判性思维、架构演进意识

**回答要点**：
- Agent 当前基于 `langchain_classic` 的 `AgentExecutor`（官方已标记为 legacy），计划迁移到 **LangGraph**：
  - 用 StateGraph 实现"必须先检索再回答"的硬约束，而非 Prompt 软约束
  - 用 Checkpointer 接管多轮记忆，告别手工拼接历史文本
  - 敏感操作（如发邮件）增加人机协同中断确认
- BM25 在小规模文档集上效果不佳（IDF 统计不可靠），后续按查询意图分类按需启用

## 二、RAG 检索增强生成

### Q4：RAG 的整体流程是怎样的？
**考察点**：RAG 基础原理

**回答要点**：
1. **离线阶段**：文档上传 → 解析（PDF/Word/Markdown/TXT）→ 文本分割（chunk_size=500, overlap=50）→ DashScope 向量嵌入 → 存入 Chroma
2. **在线阶段**：用户提问 → HyDE 假设文档生成 → 向量检索（DashScope Embedding）+ BM25 关键词检索（jieba 分词）→ 合并去重 → BGE-Reranker 重排序 → Top-K 上下文 → LLM 生成回答 → SSE 流式返回

### Q5：为什么要用 HyDE？它的原理是什么？
**考察点**：检索增强技术深度

**回答要点**：
- **核心问题**：用户提问往往是口语化的（"怎么用 Git"），而文档内容是正式书面语（"Git 版本控制操作指南"），直接向量检索存在语义鸿沟
- **HyDE 解法**：让 LLM 根据用户问题**先生成一段"假设文档"（Hypothetical Document）**，再用这段假设文档去向量库中检索
- **效果**：实验数据表明，仅加入 HyDE 就将 Recall@1 从 53.33% 提升到 60.00%，HyDE+Rerank 达到 66.67%
- **代价**：多一次 LLM 调用，增加约 1~2s 延迟

### Q6：混合检索是怎么做的？为什么需要混合检索？
**考察点**：检索策略设计

**回答要点**：
- **向量检索**（语义匹配）：适合模糊语义问题，如"怎么提升代码质量"
- **BM25 关键词检索**（精确匹配）：适合精确专有名词查询，如"Redis 持久化 RDB AOF"
- **Rerank 重排序**（Cross-Encoder 精排）：对合并后的候选集重新打分排序
- **合并策略**：向量检索取 top-15，BM25 取 top-10，按 `page_content` 去重后送入 Reranker 取 top-3
- **实验结论**：小规模文档集下 BM25 可能引入噪声（IDF 统计不可靠），HyDE+Rerank 是当前综合最优策略

### Q7：Chroma 和 DashScope Embedding 是怎么用的？
**考察点**：向量数据库实操

**回答要点**：
- Chroma 作为本地持久化向量库，`persist_directory` 指向 `app/data/storage/chroma_db`
- 使用阿里云 DashScope Embedding（`DashScopeEmbeddings`）进行文本向量化
- 单例模式管理 `VectorStoreService`，避免重复创建连接
- 启动时自动构建 BM25 全量索引

### Q8：文本分割的 chunk_size 和 overlap 是如何选择的？
**考察点**：文档处理经验

**回答要点**：
- `chunk_size=500`，`chunk_overlap=50`，分隔符按优先级从段落 → 换行 → 标点递进
- 500 大小平衡了语义完整性和检索精度（太小则上下文断裂，太大则检索噪声多）
- 10% 的 overlap 保证分割边界不切断关键信息
- 支持中英文混合分隔符：`["\n\n", "\n", "\t", " ", ".", ",", "!", "?", ";", "。", "，"]`

### Q9：BGE-Reranker 是怎么工作的？为什么需要它？
**考察点**：重排序模型原理

**回答要点**：
- 使用 `BGE-Reranker-Base`（本地部署的 Cross-Encoder 模型）
- **Bi-Encoder vs Cross-Encoder**：向量检索用的是 Bi-Encoder（query 和 doc 独立编码），速度快但精度有限；Reranker 是 Cross-Encoder（query+doc 联合编码），精度更高但速度慢
- **策略**：用 Bi-Encoder 粗召回（高召回），再用 Cross-Encoder 精排（高精度），兼顾效率与效果
- 选用 Base 版本而非 Large/M3 是为了**本地推理的速度与资源平衡**

### Q10：在什么情况下 BM25 检索反而降低了召回率？
**考察点**：检索策略批判性分析

**回答要点**：
- 实验显示：纯模糊语义场景下，HyDE+Rerank 的 Recall@1=66.67%，加入 BM25 后降至 50.00%
- **原因**：
  1. 小规模文档集（13 篇）下 **IDF 统计不可靠**，无法有效区分专有词与普通词
  2. 语义模糊问题缺乏精确关键词，BM25 匹配到的多是噪声文档
  3. 去重合并时噪声文档挤占了真正相关文档的排名位置
- **BM25 的适用前提**：① 文档规模足够大；② 查询包含明确专有词汇

## 三、LangChain Agent 与 Function Calling

### Q11：Agent 是怎么设计的？有哪些工具？
**考察点**：Agent 架构设计

**回答要点**：
- 基于 `langchain_classic` 的 `create_tool_calling_agent` + `AgentExecutor` 封装
- **6 个 Function Calling 工具**：
  1. `search_knowledge_base`：知识库检索（HyDE+向量+BM25+Rerank 全流程）
  2. `upload_document`：文档上传入库
  3. `get_document_status`：知识库统计 + 关键词过滤
  4. `generate_report`：检索 + LLM 汇总 → Markdown 报告
  5. `convert_format`：报告格式转换（md/txt/docx）
  6. `send_email`：SMTP 邮件发送 + 附件
- Prompt 中通过**核心原则**硬约束：默认先检索知识库再回答，避免 Agent 凭训练知识直接回答

### Q12：Agent 的 Prompt 是怎么设计的？
**考察点**：Prompt Engineering

**回答要点**：
- 三层结构：
  1. **角色定义**："你是一个企业知识库助手"
  2. **核心原则**："所有用户提问，默认先调用 search_knowledge_base 检索知识库"
  3. **例外规则**：纯社交用语、数学计算、用户明确说不用查文档
- 每个工具有明确的**适用/不适用场景说明**
- 回答格式约定："根据知识库文档：" vs "根据通用知识："
- 冲突处理：知识库内容优先于通用知识

### Q13：AgentExecutor 的流式输出是怎么处理的？
**考察点**：Agent 流式输出机制

**回答要点**：
- 使用 `chain.astream()` 异步流式调用
- 解析三种 chunk 类型：
  - `"actions"`：Agent 正在调用工具，推送 `[调用工具: xxx]` 提示
  - `"steps"`：工具执行结果。检测 `[REPORT_FILE]` 标记自动推送下载按钮 HTML
  - `"output"`：最终文本回复，逐 token 推送
- SSE 协议：`data: {content}\n\n` 格式，换行符需转义为 `\\n`
- 流式结束后持久化对话历史：1 条 HumanMessage + 1 条 AIMessage

### Q14：LangChain 工具定义时有哪些坑？
**考察点**：LangChain 实战经验

**回答要点**：
- **多参数工具必须加 `args_schema`**：否则 LLM 可能脑补参数格式导致解析失败
- **`@tool` 装饰器使函数变为不可直接调用的对象**：需通过 `.invoke()` 或 Agent 间接调用
- **docstring 优先级高于函数名**：工具描述以 docstring 为准，函数名仅作标识
- **工具返回值建议用纯文本**：复杂 JSON 结构可能导致 Agent 解析异常
- **参数传递的 LLM 脑补陷阱**：LLM 可能将纯文本对话当作 JSON 参数传入，需在工具内做兼容处理

### Q15：为什么计划从 AgentExecutor 迁移到 LangGraph？
**考察点**：技术选型与架构演进

**回答要点**：
- `AgentExecutor` 是 LangChain 官方标记的 **legacy** 方案，存在三个核心局限：
  1. **流程控制只能靠 Prompt 软约束**：无法强制"先检索后回答"，模型可能跳过检索
  2. **对话历史需手工拼接注入**：每次请求手动格式化历史文本拼到 input 中
  3. **非真正的 token 级流式**：Agent 思考→工具调用→回答是分阶段返回的
- **LangGraph 优势**：
  - StateGraph 图结构**硬约束**流程（入口强制经过检索节点）
  - Checkpointer 自动管理多轮对话状态
  - 真 token 级流式输出
  - 支持人机协同中断（如发邮件前等待确认）

## 四、消息队列与异步处理

### Q16：为什么需要 RabbitMQ？文档上传的异步流程是怎样的？
**考察点**：消息队列设计

**回答要点**：
- **同步上传的问题**：大文件解析 + 向量化耗时长（可能 10~30s），用户需等待，接口超时风险高
- **异步方案**：
  1. 用户上传文件 → API 将文件内容写入 Redis（临时存储，600s 过期）
  2. 发布消息到 RabbitMQ Topic 交换机 → 立即返回 HTTP 202 + task_id
  3. 内嵌 Worker 异步消费消息 → 解析 → 向量化 → 更新 Chroma + BM25 索引
  4. 用户通过 task_id 轮询任务状态（Pending → Processing → Completed/Failed）
- **可靠性保障**：消息持久化（`delivery_mode=PERSISTENT`）、死信队列（DLX）、幂等性检查（Redis 记录已处理 task_id）、重试机制（最多 3 次，间隔 5s）

### Q17：RabbitMQ 消息的可靠性和幂等性是怎么保证的？
**考察点**：消息队列高级特性

**回答要点**：
- **可靠性**：
  - 消息持久化：`DeliveryMode.PERSISTENT`，队列 `durable=True`
  - 死信队列：`x-dead-letter-exchange` + `x-dead-letter-routing-key`，处理失败消息
  - 手动 Ack：`async with msg.process()`，处理成功才确认
  - 重试机制：消费失败后最多重试 3 次，间隔 5 秒
- **幂等性**：
  - Redis key `task:processed:{task_id}` 记录已处理任务（24h 过期）
  - 消费前检查：若已处理则直接跳过

### Q18：RabbitMQ 连接失败时系统怎么处理？
**考察点**：容错设计

**回答要点**：
- FastAPI 的 `lifespan` 中 RabbitMQ 连接是**可选**的：连接失败时日志告警但不阻塞服务启动
- 文档异步上传功能降级禁用，其他功能（检索问答、报告生成等）正常可用
- 连接成功后自动启动内嵌 Worker 消费

## 五、缓存策略

### Q19：双层缓存是怎么设计的？
**考察点**：缓存架构

**回答要点**：
- **第一层：语义相似度缓存**（内存向量，Faiss-like）
  - 将历史问题的 Embedding 向量存在内存中
  - 新问题先编码 → 与缓存向量计算余弦相似度
  - 阈值 0.85，命中则直接返回缓存答案
  - 优点：能匹配"换一种问法"的相似问题
- **第二层：MD5 精确匹配缓存**（Redis）
  - 对问题文本做 MD5 哈希
  - 命中则直接返回（完全相同的提问）
  - Redis 过期时间 600s
- **效果**：缓存命中时延迟从 ~20s 降至 ~0.01s

### Q20：语义缓存的向量相似度是怎么计算的？
**考察点**：向量计算

**回答要点**：
- 使用 `DashScopeEmbeddings` 对问题文本编码 → 归一化的 1024 维向量
- 余弦相似度通过向量点积计算：`np.dot(matrix, query_vec)`
- 阈值设为 0.85（经验值，平衡命中率和准确性）
- 每次对话结束后同时写入 MD5 缓存和语义缓存

### Q21：Redis 不可用时影响哪些功能？
**考察点**：降级策略

**回答要点**：
- Redis 是**懒加载降级**的：连接失败不阻塞启动
- 受影响：双层缓存不可用（不影响核心问答）、文档异步上传不可用（文件内容通过 Redis 传递）
- 不受影响：知识库检索问答、报告生成、对话历史持久化（基于文件）

## 六、认证与安全

### Q22：JWT 认证是怎么实现的？
**考察点**：认证机制

**回答要点**：
- 使用 `python-jose` 库实现 JWT（HS256 算法）
- 密码使用 `passlib` 的 `pbkdf2_sha256` 哈希存储
- Token 有效期 7 天
- `HTTPBearer` + `Depends(get_current_user)` 依赖注入验证
- 多用户数据物理隔离：对话历史按 `user_id` 分目录存储

### Q23：接口限流是怎么做的？
**考察点**：安全防护

**回答要点**：
- 使用 `slowapi` 库 + `SlowAPIASGIMiddleware` 中间件
- 聊天接口限制 **10 次/分钟**（基于客户端 IP）
- 可配置开关：`RATE_LIMIT_ENABLED` 环境变量控制
- 超限返回 429 状态码

## 七、工程化与部署

### Q24：项目是怎么处理多环境配置的？
**考察点**：配置管理

**回答要点**：
- 使用 `pydantic-settings` + `.env` 文件管理配置
- `SettingsConfigDict` 自动加载 `.env`，`extra="ignore"` 忽略未定义字段
- 环境区分：`ENV` 字段支持 dev/staging/production
- 生产环境全局异常处理返回通用错误信息（不暴露内部细节）
- `.env.example` 模板文件供新开发者参考

### Q25：FastAPI 的 lifespan 中做了哪些初始化？
**考察点**：应用生命周期管理

**回答要点**：
- **启动时**：
  1. RabbitMQ 连接 + 内嵌 Worker 启动（失败不阻塞）
  2. BM25 全量索引构建
  3. SQLite 表自动创建
- **关闭时**：取消 Worker 任务 + 断开 RabbitMQ 连接
- 使用 `@asynccontextmanager` 实现优雅启停
- 日志系统在模块加载时初始化

### Q26：SSE 流式对话的实现细节？
**考察点**：服务端推送

**回答要点**：
- FastAPI `StreamingResponse` + `media_type="text/event-stream"`
- 前端使用 `EventSource` API 接收
- 换行符转义：`\n` → `\\n`，防止破坏 SSE `data:` 格式
- 流结束标记：`data: [DONE]\n\n`
- 前端使用 `marked.js` 实时渲染 Markdown（表格、代码块、标题等）
- 流式结束后统一持久化对话历史（HumanMessage + AIMessage 双次写入）

### Q27：项目是怎么做检索效果评估的？
**考察点**：质量保障

**回答要点**：
- 构建了手工测试集 `eval_questions.json`（30 条模糊 + 13 条精确问题）
- 每条测试包含 `question` + `expected_keywords`
- 评估指标：
  - **Recall@K**：Top-K 结果中是否包含预期关键词
  - **MRR**（Mean Reciprocal Rank）：第一个正确答案排名的倒数均值
- 自动化脚本 `eval_retrieval.py` 支持多种策略 A/B 对比
- 实验结论写入 `EVALUATION.md`：HyDE+Rerank 在纯模糊场景下 Recall@1 达 66.67%

### Q28：Docker 是怎么用的？
**考察点**：容器化部署

**回答要点**：
- 项目包含 `Dockerfile` 和 `docker-compose.yml`
- Redis 通过 Docker 运行：`docker run -d -p 6379:6379 redis`
- RabbitMQ 在 VM 中通过 Docker 运行（持久化数据卷挂载）
- 注意：Docker RabbitMQ 重启后数据持久化需配置 volume

## 八、Python / FastAPI 基础

### Q29：`@lru_cache` 在项目中是怎么用的？
**考察点**：Python 装饰器

**回答要点**：
- `get_agent()` 和 `get_settings()` 使用 `@lru_cache(maxsize=None)` 实现单例
- `maxsize=None` 表示无限缓存，确保全局唯一实例
- 避免每次请求重复创建 Agent / 重新加载配置
- 注意事项：**装饰器必须加括号** `@lru_cache()` 而非 `@lru_cache`

### Q30：FastAPI 的依赖注入（Depends）在项目中怎么用的？
**考察点**：FastAPI 核心特性

**回答要点**：
- `Depends(get_current_user)` 从 HTTP Bearer Token 中解析用户信息
- 注入到路由函数参数中，自动完成认证校验
- 无效 Token 自动抛出 401 异常

### Q31：`Pydantic` 和 `pydantic-settings` 的区别？
**考察点**：库选型理解

**回答要点**：
- `pydantic`：数据校验和序列化（Request/Response Schema）
- `pydantic-settings`：专门用于配置管理，从 `.env` 文件和环境变量加载配置，类型安全 + 自动校验

## 九、开放性问题

### Q32：如果你要重新设计这个项目的检索链路，你会怎么做？
**考察点**：系统设计能力

**参考思路**：
- 引入**查询意图分类**：先判断是模糊语义问题还是精确关键词问题，按需启用 BM25
- 考虑**多路召回融合**（RRF 算法）替代简单的拼接去重
- 引入**RAPTOR / 层次化索引**处理长文档的跨段落检索
- 考虑**Self-RAG**或**Corrective RAG**：检索后自我评估相关性，不足时自动补充检索

### Q33：如果有 100 万篇文档，当前的架构哪些地方会成为瓶颈？
**考察点**：架构扩展性

**参考思路**：
- Chroma 本地单机存储无法支撑，需迁移到分布式向量库（Milvus/Qdrant）
- BM25 全量索引重建开销大，需增量索引 + 定期全量重建
- DashScope Embedding API 调用频率受限，需批量处理 + 限速
- 语义缓存内存向量会 OOM，需引入向量数据库存储缓存
- Reranker 本地推理吞吐有限，需考虑 GPU 加速或多模型并行

### Q34：项目中遇到的最棘手的技术问题是什么？怎么解决的？
**考察点**：问题解决能力

**参考思路**：
- Agent 工具参数传递的 LLM 脑补问题：LLM 会将对话文本当作 JSON 参数传入工具
  - 解决：工具内做 JSON 与纯文本兼容解析 + 添加 `args_schema` 严格约束参数格式
- `@tool` 装饰器使函数变为不可直接调用对象，导致测试困难
  - 解决：理解 LangChain Tool 的调用机制，通过 `.invoke()` 间接调用
- 小模型使用 ReAct 格式时输出不稳定
  - 解决：切换到 Function Calling 模式 + 明确 Prompt 约束
